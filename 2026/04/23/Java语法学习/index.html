IOC（控制反转、Spring、DI）

IOC是思想, DI是实现方式 ,将自己可以new的对象交给容器管理 ,容器在我们需要对象的时候DI
作用：

  1. 对象之间的耦合度或者说依赖程度降低。
  2. 资源变的容易管理；比如你用 Spring 容器提供的话很容易就可以实现一个单例。

IoC 容器

容器就是存放和管理 Bean 的地方

Bean 的作用域

作用域 说明 使用场景
singleton（默认） 整个容器只有一个实例 无状态服务
prototype 每次获取都创建新实例 有状态对象
request（Web） 每个请求一个实例 Web应用
session（Web） 每个会话一个实例 用户会话

三种依赖注入:

  • 1️⃣ 字段注入（最常用）
  • 2️⃣ 构造器注入（推荐，官方推荐）
  • 3️⃣ Setter 注入（用于可选依赖）

核心注解详解

1. @Component - 将类标记为 Bean
2. @Autowired - 自动注入依赖
3. @Configuration - 配置类
4. @Value - 注入配置值

Aop 面向切面编程

AOP（面向切面编程）的核心作用

AOP 允许你将横切关注点（跨越多个模块的公共逻辑）从业务代码中抽取出来，实现关注点分离。

一、主要作用总结

作用 说明 典型场景
日志记录 统一记录方法调用、参数、返回值 审计、调试
性能监控 统计方法执行时间 慢查询排查、性能瓶颈分析
事务管理 统一管理数据库事务 增删改操作的开启/提交/回滚
权限校验 方法调用前检查用户权限 接口鉴权、角色验证
缓存管理 方法返回值缓存 减少数据库查询
异常处理 统一捕获和处理异常 错误日志、降级处理
参数校验 统一验证输入参数 接口入参合法性
幂等性控制 防止重复提交 表单重复提交、接口重试

三、AOP 的执行时机（通知类型）

类型 注解 执行时机
前置通知 @Before 方法执行前
后置通知 @After 方法执行后（无论是否异常）
返回通知 @AfterReturning 方法正常返回后
异常通知 @AfterThrowing 方法抛出异常后
环绕通知 @Around 方法执行前后（最强大）

五、实际应用场景总结

场景 是否必须用 AOP 替代方案
全局日志 ✅ 强烈推荐 手动在每个方法写日志 ❌
事务管理 ✅ Spring 已内置 手动开启/提交 ❌
权限校验 ✅ 推荐 在每个 Controller 写校验 ❌
接口限流 ✅ 推荐 手动判断 ❌
数据脱敏 ✅ 推荐 在每个 getter 处理 ❌
分布式锁 ✅ 推荐 手动加锁/释放 ❌

一句话总结：AOP 让你不要在业务代码中重复写日志、事务、权限等样板代码，把这些公共逻辑抽到切面里统一管理，业务代码只关心核心逻辑。
